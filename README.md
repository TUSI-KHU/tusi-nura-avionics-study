# TUSI-NURA Avionics Study Guide

이 저장소는 **초보자도 단계별 TODO를 따라가며 비행 컨트롤러 상태머신을 완성**하도록 설계된 학습 코스입니다.

## 1) 목표
- C/Arduino 기초 문법으로 상태 기반 제어를 구현한다.
- 센서(IMU, BARO) 데이터를 읽고 전이 조건으로 활용한다.
- SD 로그를 통해 동작을 검증 가능한 형태로 남긴다.
- 최종적으로 `BOOT -> SAFE -> ARMED -> FLIGHT -> RECOVERY` 흐름을 1회 이상 재현한다.

## 2) 저장소 구조
- `problems/Pxx_.../template.ino`: 학습자가 작성할 템플릿
- `problems/Pxx_.../problem.md`: 문제 설명/요구사항
- `answers/Pxx_.../answer.ino`: 7문제 기준 참고 정답(자력 구현 후 비교 권장)

권장 순서: **P01 -> P07(압축 코스)** 순차 진행

## 3) 하드웨어/기본 연결 체크
학습 시작 전에 아래 하드웨어 연결을 먼저 점검하세요.

- LED: `D8`
- Button: `D4` (`INPUT_PULLUP` 기준: 눌림=LOW)
- Buzzer: `D9`
- SD module CS: `D10`
- I2C: `A4=SDA`, `A5=SCL`
- MPU6050: I2C 연결
- BMP180: I2C 연결

## 4) 라이브러리 체크리스트
아래 라이브러리가 준비되어 있어야 합니다.

- `Wire.h`
- `SPI.h`
- `SD.h`
- `Adafruit_MPU6050.h`
- `Adafruit_Sensor.h`
- `Adafruit_BMP085.h`

## 5) 학습 규칙 (중요)
- 각 문제는 **template.ino의 TODO 순서대로**만 진행하세요.
- 한 TODO를 끝낼 때마다 바로 업로드/실행 검증하세요.
- 문제를 건너뛰지 마세요. 특히 압축 코스 `P02~P05` 기반이 `P06~P07`에 직접 연결됩니다.
- 정답 파일은 막힐 때 비교용으로만 사용하세요.

## 6) 7문제 압축 커리큘럼 (난이도 균등 상승)

난이도 점프를 고르게 만들기 위해, 각 단계에서 **새 개념 1~2개만 추가**하도록 구성했습니다.

### P01: 상태/입출력 기초
- 핵심: `BOOT->SAFE`, 버튼 롱프레스, `SAFE/ARMED` 전환, LED/버저 피드백
- 추가 사고 요소: 시간 기반 상태 전이 + 롱프레스 타이머 기반 양방향 전환

통과 기준:
- 부팅 3초 점멸 후 SAFE 진입
- 3초 이상 누르면 SAFE/ARMED가 바뀌고, 계속 누르면 3초마다 다시 바뀜
- 상태 변경 시 버저 1회, LED 표시가 상태와 일치

### P02: 관측 가능성 + IMU 입력
- 핵심: 100ms 디버그 출력, MPU6050 초기화/50ms 읽기
- 추가 사고 요소: 비차단 주기 스케줄링 2개(100ms, 50ms) 동시 운용

통과 기준:
- `[DBG] t=... state=... btn=...` 형식 고정 출력
- IMU 초기화 실패 시 `[ERR]` 후 정지
- `ax, ay, az`가 흔들림에 따라 유의미하게 변화

### P03: 이벤트 검출 + FLIGHT 진입
- 핵심: `mag` 계산, 임계치 + 유지시간으로 `ARMED->FLIGHT`
- 추가 사고 요소: 조건 유지 타이머(`launchStartMs`) 설계

통과 기준:
- `mag`와 `event`를 주기적으로 출력
- `mag > THRESHOLD`가 `HOLD_MS` 유지될 때만 FLIGHT 진입
- 조건 해제 시 유지 타이머가 올바르게 초기화

### P04: 고도 기반 사고 확장
- 핵심: `alt0` 캘리브레이션, `relAlt`, 시간 기반 `FLIGHT->RECOVERY`
- 추가 사고 요소: 센서 기준점 보정 + 전이 reason 기록

통과 기준:
- 시작 후 기준고도 평균(`alt0`) 계산 완료
- `relAlt`가 시작점에서 0 근처
- FLIGHT 10초 후 `RECOVERY(reason=TIME)` 전이

### P05: 코어 상태머신 통합(시간 RECOVERY)
- 핵심: `BOOT->SAFE->ARMED->FLIGHT->RECOVERY` 단일 코드 통합
- 추가 사고 요소: 함수 분리 + 루프 순서 고정(입력->센서->전이->출력)

통과 기준:
- 상태머신 `BOOT->SAFE->ARMED->FLIGHT->RECOVERY` 1회 완주
- `ARMED->FLIGHT`가 IMU 임계치+유지시간으로 안정 전이
- `FLIGHT->RECOVERY`가 시간 조건(`TIME`)으로 안정 전이
- loop 순서(입력->센서->전이->출력)가 일관됨

### P06: 고도 기반 RECOVERY 정밀화
- 핵심: `maxRelAlt` 추적 + 고도 하강량 조건 추가
- 추가 사고 요소: 다중 조건 우선순위(ALT 우선, TIME 백업)

통과 기준:
- `FLIGHT->RECOVERY`에서 `ALT` 조건이 먼저 검사됨
- `ALT` 불충족 시 `TIME` 백업 조건이 동작
- reason 로그가 `ALT`/`TIME`으로 정확히 구분됨

### P07: SD 로그 계층 추가(실패 격리)
- 핵심: SD 초기화, CSV 헤더 1회 작성, 100ms 주기 append
- 추가 사고 요소: SD 실패를 비치명으로 분리하는 `logEnabled` 설계

통과 기준:
- `flight.csv` 생성 및 헤더 1회 작성
- 컬럼 순서 `millis,state,accMag,relAlt` 고정
- SD 실패 시 로깅만 비활성화되고 루프는 계속 동작

## 7) 문제 풀이 방법(권장)
1. 해당 `template.ino` 열기
2. TODO 1만 구현
3. 업로드 후 동작 확인
4. 통과하면 각 TODO 구현 후 업로드 및 테스트 반복
5. 마지막 TODO까지 완료 후 해당 문제의 통과 기준으로 동작을 검증

권장 기록:
- 무엇을 바꿨는지
- 기대 동작 vs 실제 동작
- 실패 원인/수정 내용

## 8) 통합 단계 디버깅 우선순위
특히 압축 코스 P06/P07에서 아래 순서로 디버깅하세요.

1. **상태 전이 로그**: 전이가 언제/왜 일어났는지 먼저 확인
2. **LED/버저 표시**: 상태 표시가 실제 state와 일치하는지 확인
3. **센서 값**: `accMag`, `relAlt`, `maxRelAlt`가 정상 범위인지 확인
4. **SD 로그**: 헤더/주기/컬럼 순서 확인

앞 단계가 깨지면 뒤 단계 점검은 보류하고 먼저 앞 단계를 고치세요.

## 9) 자주 발생하는 문제와 해결 방향
- 버튼이 반대로 동작: `INPUT_PULLUP` 논리(LOW=눌림) 재확인
- 버튼 전환이 너무 빠름/느림: 3초 비교식과 `pressStartMs`를 언제 갱신하는지 확인
- 센서 초기화 실패: 전원/GND/I2C 배선 재확인
- 전이가 너무 빠름/느림: 임계치, HOLD_MS, 타이머 초기화 위치 확인
- SD는 되는데 상태가 멈춤: SD 실패를 치명 에러로 처리하지 않았는지 확인
