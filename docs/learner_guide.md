# 미니 Flight Computer 학습 가이드

이 문서는 강의자용 운영안이 아니라, 학습자용 안내 문서입니다.
목표는 "문제를 순서대로 풀면서" 아두이노 기반 미니 Flight Computer를 완성하는 것입니다.

## 1. 이 자료를 어떻게 쓰면 되나요?

각 문제 폴더에는 3개 파일이 있습니다.

- `problem.md`: 오늘 할 것
- `template.ino`: 시작 코드
- `answer.ino`: 막혔을 때 비교할 코드

권장 순서:
1. `problem.md` 읽기
2. `template.ino`의 TODO 채우기
3. 업로드/테스트
4. 안 되면 시리얼 로그 확인
5. 마지막에 `answer.ino`와 비교

## 2. 전체 학습 순서 (P01 ~ P14)

1. P01: LED로 BOOT/SAFE 상태 표시
2. P02: 버튼 3초 누름으로 SAFE/ARMED 변경 + 버저 1회
3. P03: 상태를 시리얼로 주기 출력
4. P04: MPU6050 값 읽기
5. P05: 가속도 크기 계산 + 임계치 비교
6. P06: ARMED -> FLIGHT 전이
7. P07: BMP180 상대고도 계산
8. P08: 시간 기반 RECOVERY 전이
9. P09: 고도 기반 RECOVERY 보강
10. P10: 상태/센서/전이 통합 (SD 제외)
11. P11: SD 카드 기본 동작 확인
12. P12: SD CSV 로그 저장
13. P13: 통합 코드에 SD 결합
14. P14: 전체 동작 점검

## 3. 왜 상태머신이 중요한가?

FC는 상황에 따라 해야 할 일이 달라집니다.

- SAFE: 대기
- ARMED: 발사 감시
- FLIGHT: 비행 중 판단
- RECOVERY: 종료 단계

상태머신을 쓰면 좋은 점:
- 지금 어떤 모드인지 `state` 하나로 명확해짐
- 조건 충돌이 줄어듦
- 디버깅이 쉬워짐

한 줄로 말하면:
"코드를 기능별로 짜는 게 아니라, 상태별로 나누면 실수가 줄어든다."

## 4. 왜 `delay()`를 줄이고 `millis()`를 쓰나?

`delay(1000)`을 쓰면 1초 동안 다른 일을 못 합니다.

문제:
- 버튼 입력 놓침
- 센서 갱신 끊김
- 로그 주기 깨짐

`millis()` 방식은 루프를 계속 돌리면서
"시간이 됐을 때만 해당 작업"을 실행합니다.

예:
- 100ms마다 로그 출력
- 50ms마다 IMU 읽기
- 매 루프마다 상태 전이 확인

## 5. 초심자가 먼저 익힐 C/아두이노 최소 문법

- `if (...) { ... }`: 조건문
- `=` vs `==`: 대입/비교
- `&&`, `||`, `!`: 논리 연산
- `unsigned long`: 시간값(`millis`) 저장
- 함수 분리: `readIMU()`, `updateState()`처럼 역할 나누기

이 5개만 정확히 이해해도 문제를 끝까지 따라갈 수 있습니다.

## 6. 막혔을 때 체크 순서

1. 배선 확인 (VCC/GND/핀 번호)
2. 포트/보드 설정 확인
3. 시리얼 baudrate 115200 확인
4. 시리얼에 값이 나오는지 먼저 확인
5. 상태 변수(`state`)가 바뀌는지 확인

코드보다 배선/설정 문제인 경우가 더 많습니다.

## 7. 학습 목표

이 자료의 목표는 "고급 알고리즘"이 아닙니다.

- 상태를 나누어 설계하기
- 센서 값을 읽고 조건으로 상태 바꾸기
- 동작을 로그로 남기기

즉, 임베디드 프로젝트의 기본 흐름을 몸에 익히는 것이 목표입니다.
